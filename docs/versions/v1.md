# QuestionaireRAG — Version 1

**Date:** 2026-03-30
**Status:** POC Planning

---

## Problem Statement

Given a set of customer documents (SOPs / SOWs), retrieve and rank the most relevant **individual questions** from a dataset of questionnaire PDFs.

---

## Requirements

### Inputs
- **Questionnaire PDFs** — indexed once, reused across queries. Each PDF contains questionnaires; parsed down to individual questions.
- **Customer PDFs** — one or multiple SOP/SOW docs per customer, uploaded at query time.

### Output
- Ranked list of individual **questions** most relevant to the customer's documents.
- Each result includes: question text, source document, page, relevance score.

### Matching Strategy
- **Hybrid** — semantic (dense vector) + keyword (BM25 sparse) combined via Reciprocal Rank Fusion (RRF).

### Scale
- POC: dozens of questionnaire docs.
- Future: designed to scale to thousands.

---

## Architecture (v1)

### Indexing Pipeline (one-time / on new questionnaire upload)
1. Parse questionnaire PDFs → extract individual questions (via LLM)
2. Embed each question → dense vector
3. Index keywords per question → BM25 sparse index
4. Store in ChromaDB with metadata: `{ source_doc, page, question_id, category }`

### Query Pipeline (per customer)
1. Parse customer SOP/SOW PDFs → extract text
2. LLM summarizes key topics/intent from customer docs
3. Run hybrid search:
   - Dense: semantic similarity via embeddings
   - Sparse: BM25 keyword match
4. Reciprocal Rank Fusion (RRF) to merge ranked results
5. Return ranked list of questions via FastAPI endpoint

Embedding Model — all-MiniLM-L6-v2
What it does: Converts text into a list of numbers (a vector) that captures the meaning of the text. Similar sentences end up with similar vectors.

Why we used it: It's free, runs locally (no API calls), fast, and good enough for semantic similarity tasks. The "MiniLM" part means it's a small/efficient model — 384 numbers per sentence vs 1536 for OpenAI's model.

LLM — llama-3.1-8b-instant via Groq
What it does: Reads the customer's SOP/SOW PDFs and writes a short summary of what the company cares about (e.g. "this company focuses on data encryption, access control, and HIPAA compliance").

Why we used it: That summary becomes the search query — instead of dumping raw PDFs into search, we let the LLM extract the key ideas first. Groq gives a free API with very fast inference.

ChromaDB
What it does: A database that stores and searches vectors. You give it a query vector and it returns the most similar ones (cosine similarity).

Why we used it: Handles the semantic half of our search — finds questions that are conceptually close to the customer summary even if they use different words.

BM25 (rank_bm25)
What it does: Classic keyword search — scores documents by how often your query words appear in them, weighted by rarity (common words like "the" count less).

Why we used it: Catches exact keyword matches that semantic search misses. E.g. if the customer says "SOC 2", BM25 will reliably surface questions with "SOC 2" in them.

Reciprocal Rank Fusion (RRF)
What it does: Takes two ranked lists (one from ChromaDB, one from BM25) and merges them into a single ranked list fairly — using rank position, not raw scores.

Why we used it: ChromaDB and BM25 scores are on completely different scales and can't be directly compared. RRF sidesteps that by only caring about rank order, not the actual score values.

PyMuPDF (fitz)
What it does: Reads PDF files and extracts the raw text from them.

Why we used it: Fastest and most reliable Python PDF parser. The customer docs are PDFs, so we need this to get any text out of them at all.

openpyxl
What it does: Reads Excel (.xlsx) files row by row.

Why we used it: The CAIQ questionnaire comes as an XLSX file. This library lets us pull out each question, its ID, and its domain cleanly.

FastAPI
What it does: A Python framework for building REST APIs. Handles HTTP requests, routing, and auto-generates Swagger docs at /docs.

Why we used it: Lets us expose the indexing and query logic as endpoints so any frontend or tool can call it. The auto-generated /docs page is useful for testing without writing a frontend.
