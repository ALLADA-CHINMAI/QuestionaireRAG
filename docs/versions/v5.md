# QuestionaireRAG — Version 5

**Date:** 2026-04-14
**Status:** Web UI + Fresh Embeddings + Bug Fixes
**Version Codename:** Self-Service Era

---

## Overview

**Version 5** adds a browser-based UI, removes all Chroma DB dependencies, fixes a runtime search bug, and introduces a script to create the Azure Cognitive Search index programmatically when the Azure Portal is blocked by private network restrictions.

### Key Improvements

- **Web UI**: Upload customer documents and run queries entirely from the browser — no curl or API client needed
- **Questionnaire re-indexing from UI**: Admins can upload a new CAIQ XLSX and re-index without dev access
- **Fresh embeddings**: Index always generates new Ada-002 embeddings directly from the CAIQ XLSX; stale Chroma DB embeddings are gone
- **XLSX customer docs**: Customer documents can now be Excel files, not just PDFs
- **Index creation script**: Create the `caiq_questions` index via SDK when the portal is inaccessible (private network)
- **Bug fix**: Removed unsupported `query_language` parameter that caused 500 errors on every search

### Breaking Changes

None. All v4 API endpoints remain identical.

## What Changed from v4

### 1. Bug Fix — `query_language` parameter removed (`app/core/azure_search.py`)

The `search_hybrid()` method was passing `"query_language": "en-us"` to `client.search()`. The `azure-search-documents` Python SDK does not expose this as a keyword argument, causing every search request to fail with:

```
Session.request() got an unexpected keyword argument 'query_language'
```

**Fix**: Removed the `"query_language"` key from `search_params`. Language detection is automatic and search quality is unaffected.

---

### 2. Web UI (`app/static/index.html` — new file, `main.py`)

A single-page HTML interface served at `GET /` with two steps:

**Step 1 — Index Questionnaire**
- Drag-and-drop XLSX upload zone
- "Index Questionnaire" button calls `POST /index/upload-questionnaire`
- Status banner loaded from `GET /health` on page load (shows current question count)
- Confirmation dialog if re-indexing over an existing index
- Animated loading feedback during indexing

**Step 2 — Upload Customer Docs**
- Disabled and greyed out until questionnaire index is confirmed ready
- Drag-and-drop multi-file upload zone (PDF and XLSX)
- Top K input (1–263) and semantic ranking toggle
- Calls `POST /upload-and-query`
- Results rendered as a ranked table with domain badges and score bars

`main.py` changes:
- Mounts `app/static/` as `/static` via `StaticFiles`
- Adds `GET /` route returning `FileResponse("app/static/index.html")`

### 3. New API Endpoints (`app/api/routes.py`)

#### `POST /index/upload-questionnaire`

Accepts a CAIQ XLSX file upload, generates fresh embeddings, and rebuilds the index. Intended for admin re-indexing without server access.

```
Content-Type: multipart/form-data
Body: file=<.xlsx>
```

- Validates `.xlsx` extension
- Saves to `/tmp/caiq_upload_<uuid>.xlsx`
- Calls `build_index()` to generate embeddings and upload to Azure
- Cleans up temp file in `finally` block

**Response**:
```json
{
  "message": "Indexed 'CAIQv4.0.3.xlsx' successfully",
  "questions_indexed": 263
}
```

#### `POST /upload-and-query`

Accepts one or more customer document files (PDF or XLSX), runs the full RAG pipeline, and returns ranked CAIQ questions — all in a single request. Designed for the UI.

```
Content-Type: multipart/form-data
Body: files[]=<file1>, files[]=<file2>, top_k=20, use_semantic_ranking=true
```

- Validates file extensions (only `.pdf` and `.xlsx` accepted)
- Saves files to `data/customers/upload_<uuid>/` (temporary session)
- Calls `retrieve_for_customer()` with the session ID
- Cleans up the temp directory in `finally` block regardless of outcome

**Response**: Same `QueryResponse` schema as `POST /query`.

#### `GET /health` (updated)

Now returns the actual indexed question count by reading `data/questions_store.json`, so the UI can show whether the index is populated.

```json
{
  "status": "ok",
  "index_built": true,
  "questions_indexed": 263
}
```

---

### 4. XLSX support for customer documents (`app/core/ingestor.py`)

`load_customer_docs()` previously only read PDF files. It now also reads `.xlsx` files using `openpyxl`:

- Iterates all sheets
- Joins non-empty cell values within each row with ` | `
- Appends XLSX content to the same combined text string as PDFs
- Raises `FileNotFoundError` if no PDF or XLSX files are found (previously raised only on missing PDFs)

---

### 5. Fresh embeddings — Chroma DB removed (`scripts/migrate_caiq_to_azure.py`, `requirements.txt`)

In v4, `migrate_caiq_to_azure.py` loaded pre-existing embeddings from Chroma DB and re-uploaded them to Azure. This meant embeddings could be stale if the Chroma data was old.

In v5, the script simply calls `build_index(xlsx_path)`, which parses the CAIQ XLSX and generates brand-new Ada-002 embeddings via Azure OpenAI every time. Chroma DB is no longer a dependency at any point in the pipeline.

`requirements.txt` — removed:
- `chromadb>=0.5.3`
- `rank-bm25==0.2.2`

---

### 6. Index creation script (`scripts/create_index.py` — new file)

When the Azure Portal cannot reach the search service (private network / public access disabled), the index schema cannot be created through the portal UI. This script creates the `caiq_questions` index programmatically using `SearchIndexClient` from the `azure-search-documents` SDK.

Creates the index with:
- All required fields (`id`, `question_id`, `domain`, `question_text`, `source`, `doc_name`, `metadata`, `vector`)
- HNSW vector search profile (cosine, 1536-dim, m=4, efConstruction=400, efSearch=500)
- Semantic configuration with `question_text` as content field and `domain`/`question_id` as keyword fields

Usage:
```bash
python scripts/create_index.py
```

Reads `AZURE_SEARCH_ENDPOINT`, `AZURE_SEARCH_API_KEY`, and `AZURE_SEARCH_CAIQ_INDEX_NAME` from `.env`. Uses `create_or_update_index` so it is safe to re-run — it will update an existing index rather than fail.

## Troubleshooting

### Index Not Found (404)

```
The index 'caiq_questions' for service '...' was not found.
```

The index schema does not exist in Azure yet. Create it:

```bash
python scripts/create_index.py
```

Then populate it:

```bash
curl -X POST http://localhost:8000/index/questionnaires
```

### Portal Blocked — Cannot Create Index via UI

The Azure Search service has public network access disabled. Use the SDK script instead:

```bash
python scripts/create_index.py
```

If your machine is also blocked (not on the allowed network), connect to the VPN first or temporarily add your IP in Azure Portal → Networking.

### 500 on Every Search (fixed in v5)

Was caused by `query_language` parameter — fixed by removing it from `search_params` in `azure_search.py`.

When you upload and index a questionnaire from the UI, here's the complete flow:

## UI Action
1. User drags CAIQ XLSX file into the upload zone
2. Clicks "Index Questionnaire" button
3. UI sends `POST /index/upload-questionnaire` with the file

## Backend Processing (routes.py)
1. **File validation**: Checks the file has `.xlsx` extension
2. **Temporary save**: Saves uploaded file to system temp directory (e.g., `C:\Users\...\AppData\Local\Temp\caiq_upload_<random-uuid>.xlsx`)
3. **Calls `build_index()`** with the temp file path
4. **Cleanup**: Deletes the temp file (in `finally` block, so happens even if errors occur)

## Index Building (indexer.py)

**Step 1 - Parse CAIQ Excel**
- Reads the XLSX file sheet by sheet
- Extracts question data (question_id, domain, question_text, etc.)
- Returns list of 263 question dictionaries

**Step 2 - Generate Embeddings**
- Takes all 263 question texts
- Calls Azure OpenAI `text-embedding-ada-002` model
- Generates 1536-dimensional vector embeddings for each question
- Uses your Azure AD authentication (JWT token + API key)

**Step 3 - Prepare Documents**
- For each question, creates a document with:
  - `id`: Sanitized version (e.g., `A_A-01.1` instead of `A&A-01.1`) ← **Azure-compliant key**
  - `question_id`: Original ID (e.g., `A&A-01.1`) ← **Preserved for reference**
  - `domain`: e.g., "Application & Interface Security"
  - `question_text`: The actual question
  - `source`: "CAIQ"
  - `vector`: The 1536-dim embedding array

**Step 4 - Upload to Azure Cognitive Search**
- Connects to `https://acs-pgchub-dev-wus2.search.windows.net`
- Uses index name `caiq_questions`
- Calls `upload_documents()` which creates/updates all 263 documents
- Azure automatically builds:
  - **HNSW vector index** for semantic similarity search
  - **BM25 keyword index** for text matching

**Step 5 - Save Local Cache**
- Writes questions_store.json with all question metadata
- This allows fast lookups without querying Azure every time

## Response to UI
```json
{
  "message": "Indexed 'CAIQv4.0.3.xlsx' successfully",
  "questions_indexed": 263
}
```

## Time & Costs
- **Duration**: ~30-60 seconds (mostly embedding generation)
- **API Calls**: 
  - 263 embedding requests to Azure OpenAI (batched if possible)
  - 1 bulk upload to Azure Search
- **Storage**: ~2MB in Azure Search (263 documents × 1536 floats each)

The entire process is **idempotent** - you can run it multiple times safely. Each run generates fresh embeddings and replaces all documents in the index.



Here’s what happens after you upload customer documents (PDF or XLSX) in the UI:

---

### 1. Upload (UI → Backend)
- You drag and drop one or more customer files (PDF/XLSX) and click "Upload".
- The UI sends a POST request to /upload-and-query with the files.

---

### 2. File Handling (routes.py)
- The backend saves each uploaded file to a temporary session directory (e.g., data/customers/upload_<uuid>/).
- Only PDF and XLSX files are accepted; others are rejected.

---

### 3. Document Parsing (ingestor.py)
- The system reads all PDFs and XLSX files in the session directory:
  - PDFs: Extracts all text from each page (using PyMuPDF).
  - XLSX: Reads all sheets, concatenates non-empty cell values row-by-row with | separators.
- All extracted text is combined into a single string.

---

### 4. Summarization (embedder.py)
- Only the first 8000 characters of the combined text are sent to GPT-4o for summarization.
- GPT-4o returns a focused security summary (≤ 300 words) covering key topics, requirements, risks, and technologies.

---

### 5. Embedding (embedder.py)
- The summary is embedded into a 1536-dimensional vector using Azure OpenAI’s Ada-002 model.

---

### 6. Hybrid Search (azure_search.py)
- The summary vector and summary text are used to query the CAIQ questions index in Azure Cognitive Search.
- Both vector similarity (semantic) and BM25 keyword search are used to rank results.

---

### 7. Results (routes.py)
- The top-K most relevant CAIQ questions are returned to the UI, each with:
  - Question ID, domain, question text, source, and a hybrid score.

---

### 8. UI Display
- The UI displays the ranked questions in a table with scores and domain badges.

---

**Summary:**  
After upload, your documents are parsed, summarized (first 8000 chars only), embedded, and used to retrieve the most relevant CAIQ questions via hybrid search. The results are shown in the UI.
